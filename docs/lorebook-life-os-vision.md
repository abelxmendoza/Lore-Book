# LoreBook Life OS Vision

Date: 2026-06-15

Prompt: What if ChatGPT had been designed from day one as a life memory system?

Answer: It would not be a chatbot. It would be a personal memory operating system: a trusted place to capture life, understand patterns, preserve relationships, and turn lived experience into biography.

## Product Thesis

LoreBook should be the user's autobiographical intelligence layer.

It should answer:

- What happened?
- Who mattered?
- When did this change?
- What patterns keep repeating?
- What do I believe now that I did not believe before?
- What story is my life becoming?
- What should be remembered, corrected, forgotten, or written down?

## Core Loop

1. User talks naturally.
2. LoreBook responds helpfully.
3. LoreBook extracts candidate memories, events, people, places, decisions, feelings, and beliefs.
4. User can inspect and correct what was captured.
5. Verified memory updates the life graph.
6. Life graph powers future chat, timelines, relationship intelligence, and biography.
7. The system periodically surfaces what changed.

## The Four Surfaces

### 1. Chat

Role: the universal input and reflection surface.

Should do:

- accept messy natural language
- ask clarifying questions only when needed
- stream response
- cite memory sources
- offer next actions
- support edit, retry, branch, correction

Should not do:

- expose every engine decision by default
- require the user to classify the message first
- turn personal disclosure into gamified engagement

### 2. Memory

Role: the inspectable record.

Memory objects should have:

- claim text
- source evidence
- confidence
- truth state: verified, inferred, contradicted, deprecated, corrected
- subject: person, place, event, self, relationship, belief, goal
- temporal scope: moment, period, ongoing, unknown
- sensitivity
- last used
- correction history

### 3. Timeline

Role: the chronology of lived experience.

Timeline should show:

- events
- chapters
- arcs
- gaps
- parallel threads
- relationship periods
- identity shifts
- source density
- uncertainty

Timeline is not just a calendar. It is the spine of the autobiography.

### 4. Biography

Role: the composed story.

Biography should:

- generate from verified memories
- cite evidence
- preserve user voice
- show unresolved gaps
- distinguish fact from interpretation
- allow chapter-level edits
- regenerate from selected source ranges

## Life Projects

ChatGPT Projects are workspaces for long-running work. LoreBook needs Life Projects: scoped memory containers for long-running domains.

Examples:

- Family
- Career
- Love
- Health
- Creative Work
- Grief
- Identity
- LoreBook Building
- School
- Spirituality

Each Life Project should contain:

- chats
- pinned memories
- files
- project instructions
- people
- places
- timeline nodes
- biography excerpts
- open questions

Memory modes:

- Global memory: can inform the whole Life OS.
- Project memory: prioritized inside one Life Project.
- Project-only memory: cannot leak outside that project.
- Private chat: does not create memory unless explicitly saved.

## Memory Trust Architecture

LoreBook should make memory trustworthy through controls, not promises.

Required controls:

- show what was remembered
- show why it was remembered
- show where it came from
- edit memory
- correct memory
- delete memory
- forget source
- mark too sensitive
- mark outdated
- restore prior version
- temporary chat

## Conversation System

Threads should not be simple chat logs. They should be life-context workspaces.

Thread metadata:

- title
- subtitle/life domain
- summary
- people
- places
- themes
- time span
- memory delta
- unresolved questions
- source density
- branch lineage
- privacy/memory mode

Thread views:

- sidebar list
- grouped by Life Project
- pinned
- archived
- timeline view
- entity-linked view
- search result view

## Retrieval System

Retrieval should be selective, explainable, and domain-aware.

Retrieval layers:

1. Current thread.
2. Life Project.
3. Explicitly mentioned entities.
4. Timeline around relevant dates.
5. Verified core memories.
6. Related relationships and places.
7. Similar prior conversations.
8. Deprecated/corrected claims as warnings.

Every retrieved item should have:

- reason selected
- confidence
- source
- recency
- sensitivity
- whether it was actually used in the response

## AI Interaction Patterns

LoreBook should support these response modes:

- Recall: factual, source-backed answer.
- Reflection: interpretive but grounded.
- Biography: narrative prose from evidence.
- Correction: fix memory or timeline.
- Planning: future-oriented support.
- Relationship lens: social/relational analysis.
- Timeline lens: chronology and causality.
- Archivist: conservative, evidence-only.
- Challenge: gently question a belief or pattern.
- Silence: admit insufficient evidence.

The key is mode transparency. Users should know when LoreBook is recalling versus interpreting.

## Future Product Shape

The best version of LoreBook feels like:

- ChatGPT's conversational ease.
- A private archive's reliability.
- A biographer's narrative sense.
- A therapist's continuity, without pretending to be therapy.
- A historian's source discipline.
- A personal operating system's daily utility.

## Five-Year Vision

LoreBook becomes a living autobiography:

- It knows the user's life through evidence.
- It can answer questions about any period, person, or theme.
- It helps users write their story.
- It preserves changes in identity over time.
- It protects privacy by design.
- It turns memory into a navigable, correctable, exportable life graph.

## North Star Metrics

Trust:

- percentage of memory-grounded responses with sources
- correction success rate
- memory deletion confidence
- user-reported trust

Engagement:

- weekly meaningful reflections
- return-to-thread continuation rate
- search success rate
- Life Project activity

Memory quality:

- verified memory ratio
- duplicate memory rate
- contradicted claim resolution time
- source coverage

Storytelling:

- biography chapters generated
- biography edits accepted
- source-backed chapter coverage
- exported story packets

Retention:

- weekly digest opens
- relationship page revisits
- timeline gap completions
- cross-thread recall usage

## The Product To Avoid

Do not build:

- a general chatbot with extra tabs
- an astrology-style insight machine
- a journaling app with AI summaries
- a quantified-self dashboard full of shallow scores
- a social network for memories
- a black-box memory system

Build:

- a trusted autobiographical memory system
- a life graph
- a source-backed biography engine
- a relationship and timeline intelligence layer
- a private Life OS
